SLIDE 5 — Business Requirements
--------------------------------------------------------
"The core problem we're solving: retail investors don't have access to institutional-grade
prediction tools, and the tools that do exist rarely tell you whether their predictions
were actually correct. We wanted to change that.

Our goals were concrete. Sub-100 millisecond predictions — this can't be a batch job that
runs overnight. We persist every prediction alongside the actual outcome so accuracy is
publicly verifiable over time. The whole stack deploys with a single 'docker compose up'

SLIDE 7 — System Overview: Three Layers, One Pipeline
--------------------------------------------------------
"The system has three main layers.

At the base is the data layer — Postgres stores 15-minute OHLCV bars for over 500 stocks,
plus macro inputs like Treasury series. We also pre-compute all the ML features — RSI,
MACD, Bollinger Bands, lag values, return columns, calendar fields — and store them
alongside the raw bars. Postgres becomes the canonical source of truth for both training
and inference.

The middle layer is the backend. It ingests and updates that market data, computes features,
trains XGBoost models, and saves the trained artifacts to disk. Critically, training and
inference are fully decoupled — the inference service just loads a saved artifact and
scores new rows. It doesn't retrain.

The top layer is the frontend. It pulls the inference outputs from the backend and
displays them — side-by-side charts comparing predicted prices against actual market
movement, with a symbol selector and prediction overlays. This is what turns the ML
pipeline into something you can actually demo and evaluate."

--------------------------------------------------------
SLIDE 8 — System Overview: End-to-End Lifecycle
--------------------------------------------------------
"This is the full lifecycle in one picture.

Historical data and engineered features flow into Postgres.
A training job reads from that store, trains the XGBoost model, and saves the artifact
to model_artifacts/ on disk.
Later, the inference service loads that saved artifact and scores fresh feature rows —
no retraining, just fast scoring.
The frontend then fetches those predictions and visualizes them against actual price behavior.

This is important to understand: the system is not just a script that predicts.
It's a repeatable ML pipeline with a data layer, a model lifecycle, and a serving path.
Each stage is independently testable and auditable."

SLIDE 10 — Configuration Management: Version Control
--------------------------------------------------------
"We used a modified Git Flow for version control. Main is always production-ready with
version tags. Develop is the integration branch. Every feature, bug fix, and hotfix
lives in its own branch and comes in through a pull request — no direct pushes to
develop or main.

For commit messages we followed the Conventional Commits standard: feat, fix, docs,
refactor, test. This makes the git history machine-readable and gives us a clean
changelog. Change requests went through GitHub Issues with a change-request label
so every decision is tracked and reviewable."

SLIDE 12 — CI/CD Pipeline
--------------------------------------------------------
"Here's the actual pipeline. Five jobs run on every PR to main.

Backend lint runs ruff check and format. Backend type check runs mypy.
The backend test job is the most important one — it spins up a real Postgres 16
container, runs Alembic migrations against it, then runs the full pytest suite.
No mocks, no in-memory databases — real schema, real queries.
Frontend lint runs Biome, and frontend build runs the production build.
All five must pass before a PR can merge."

SLIDE 14 — Risk Management
--------------------------------------------------------
"[Fill in your risk register content here. Suggested talking points:]

Our primary technical risks were data availability — we depend on FMP as an external
data source, which has rate limits and can go down.
We mitigated this by storing everything in Postgres so inference never requires a live
API call.

The second risk was model drift — a model trained on historical data degrades over time.
We mitigated this by decoupling training from inference so the model can be retrained
and swapped without redeploying the service.

The third risk was test coverage slipping. We mitigated this by enforcing the 80% gate
in CI so it can't degrade silently."

SLIDE 15 — Design: Key API Endpoints (summary)
--------------------------------------------------------
"The API is organized into focused endpoint groups.
Health check confirms the backend is up.
The inference endpoint takes a symbol, loads the saved XGBoost artifact, and returns
a next-day prediction response. Unknown symbols return a 404, bad inputs return 400.
The simulation endpoint runs 502 symbols across 26 intraday steps entirely from
pre-computed CSVs — zero database I/O, so it's extremely fast.
The market endpoints serve the stock list and OHLC bars for the charting layer."

--------------------------------------------------------
SLIDE 16 — Design: Domain Model and Layer Structure
--------------------------------------------------------
"The domain model has three core entities: Stock, which holds the ticker metadata;
DailyPrice for the raw OHLC time series; and Prediction for the model output and
confidence score. All API responses go through Pydantic v2 schemas for strict validation.

The layer architecture is a clean stack: routers call services, services call CRUD
functions, CRUD functions go through SQLAlchemy to Postgres.

For inference specifically, there's a model loader that reads an environment variable
called ACTIVE_MODEL to select which bundle to use. We have two bundle types:
LegacyXGBBundle is a single XGBoost regressor with a meta file,
and NextDayPathBundle is 26 separate models — one per 15-minute intraday step.
The simulation module runs entirely off pre-computed CSVs."

--------------------------------------------------------
SLIDE 17 — Design: API Endpoints (simulation callout)
--------------------------------------------------------
"To highlight the scale of the simulation module — it covers 502 symbols across
26 intraday steps. That's over 13,000 data points available instantly, with zero
database queries. It all reads from pre-computed CSV files that were generated at

--------------------------------------------------------
SLIDE 18 — Code: Backend Inference Route
--------------------------------------------------------
"This is the inference route. It's straightforward by design.

The route receives a symbol, passes it uppercased to the InferenceService, and returns
a NextDayPredictionResponse. Error handling is explicit — a ValueError that mentions
'not found' becomes a 404, other ValueErrors become 400, and anything unexpected
becomes a 500 with a message. This means the API always gives the caller enough
information to know what went wrong without leaking internal stack traces."

--------------------------------------------------------
SLIDE 19 — Code: CI Pipeline
--------------------------------------------------------
"This is the backend test job in detail. It runs on ubuntu-latest and spins up a real
Postgres 16 service container with a health check so the job doesn't start until the
database is actually ready.

We use uv for dependency management — it's significantly faster than pip.
The job installs dependencies, runs Alembic migrations against the live container,
then runs pytest. The fact that we're running real migrations in CI means if a
migration breaks, we catch it in CI, not in production."

--------------------------------------------------------
SLIDE 20 — Code: Test DB Fixture
--------------------------------------------------------
"This fixture is the foundation of our integration tests. It runs at function scope,
which means every single test gets a clean database state.

Before the test runs, it deletes any pre-existing rows for the symbols under test.
Then it seeds a known set of stocks and price data. The test runs. Then teardown
runs unconditionally — even if the test throws an exception.

This pattern guarantees test isolation. Tests can't interfere with each other, and
there's no hidden state carrying over between runs."

--------------------------------------------------------
SLIDE 21 — Code: Testing the Inference Endpoint
--------------------------------------------------------
"Two tests worth highlighting here.

The inference 404 test: we hit the predict endpoint with a fake symbol and assert we
get a 404 with the string 'not found' in the detail message. This verifies that unknown
symbol errors surface cleanly to the caller.

The health check test: hits the health endpoint and asserts a 200 with a true response.
Simple, but it's a fast smoke test that the backend is up and the DB connection works.

I'll be transparent: the success-path inference test — where we seed data and assert a
valid prediction comes back — is currently a placeholder. It's documented as KB-4 in
the testing guide. We know what's needed to fix it; it requires a DB-seeded test with
the model artifact accessible."

--------------------------------------------------------
SLIDE 22 — Code: Market Service Tests
--------------------------------------------------------
"These three tests cover the market data service layer.

The first verifies that list_stocks returns a list of StockRead objects — not raw ORM
models, not dicts — and that a seeded symbol has the right name.

The second is subtle but important: get_ohlc must return Unix timestamps as integers,
not Python date objects, because the frontend charting library expects epoch seconds.
We explicitly assert the type and do a sanity check that the value is after year 2000.

The third tests the empty case: a symbol that exists in the stocks table but has no
price rows seeded should return an empty list, not an error."
